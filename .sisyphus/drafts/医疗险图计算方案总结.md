# 医疗险图计算方案总结

## 背景

根据设计图 **"知识库与图谱中心完整搭建图方案"**，项目需要建设完整的医疗知识库体系，支撑理赔审核的智能化决策。

## 现状分析

### ✅ 已有能力
| 模块 | 实现状态 |
|------|----------|
| 理赔计算引擎 | ✅ 完整（calculator.js） |
| 规则引擎 | ✅ 已支持责任判断和金额计算 |
| AI审核 | ✅ LangGraph集成 |
| 产品配置 | ✅ 保额/免赔额/赔付比例定义 |

### 📋 待建设能力
| 模块 | 现状 | 优先级 |
|------|------|--------|
| 主数据中心（6张主表） | ❌ 无 | **P0** |
| 规则知识层（5类规则） | ⚠️ 部分 | **P0** |
| 关系图谱层（5张关系表） | ❌ 无 | P1 |
| Evidence Graph | ⚠️ 需扩展 | P1 |

## 建设方案（5个阶段）

### 阶段1: 主数据中心（P0）
**文件**: `server/knowledge/`
- 6张主表模型定义（药品/诊疗项目/疾病/医院/保单规则/别名映射）
- DAO层数据访问
- ETL模块（官方文件下载/解析/清洗）
- 别名标准化服务

### 阶段2: 规则知识层（P0）
**文件**: `server/knowledge/rules/`
- 5类规则模型（诊断-药品/诊断-项目/剂量/频次/住院必要性）
- 规则引擎集成层
- 医学合理性评估服务

### 阶段3: 关系图谱层（P1）
**文件**: `server/knowledge/graph/`
- 5张关系表（疾病-药品/疾病-项目/手术组合/产品责任）
- 关系查询服务
- 图谱可视化接口

### 阶段4: Evidence Graph（P1）
**文件**: `server/knowledge/evidence/`
- 证据图模型（节点/边定义）
- 审核流程集成
- 证据可视化服务

### 阶段5: 后台管理界面（P1）
**文件**: `components/knowledge/`
- 主数据管理页（药品/项目/疾病/医院）
- 别名映射管理页
- 关系管理页
- 规则管理页
- 版本发布页

## 关键决策

### 1. 数据存储选择
**推荐**: 继续使用JSON文件存储（当前模式）
- **理由**: 与现有架构保持一致，快速启动
- **后续**: 当数据量增长时迁移到PostgreSQL

### 2. 集成策略
**方案**: 渐进式集成
- 阶段1-2完成后即可增强现有审核流程
- 阶段3-4为增值功能，可独立发布

### 3. 开发顺序
**优先级**: P0功能优先（主数据+规则层）
- 主数据是后续所有功能的基础
- 规则层可直接增强现有审核准确性
