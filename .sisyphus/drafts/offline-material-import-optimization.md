# 离线材料导入业务逻辑优化 - 需求草稿

## 原始需求
整体优化"离线材料导入"的业务逻辑。用户批量上传材料后，先全部上传至 oss（注意 oss 返回的链接的时效性，再次调用的时候需要重新获取有效 url），再异步批量进行识别，每个材料先根据配置的理赔材料种类识别类型，确定类型后，根据不同类型配置的结构化提取 schema 进行结构化提取。上传的材料支持常见办公文件类型、图片类型、视频类型、音频类型。

---

## 现有架构分析

### 当前文件处理流程
1. **前端**: `OfflineMaterialImportDialog.tsx` - 用户选择文件
2. **文件分类**: `/api/materials/classify` - 同步分类接口（但每文件单独调用）
3. **导入**: `/api/import-offline-materials` - 创建异步任务
4. **后端**: `fileProcessor.js`, `multiFileAnalyzer.js` - 处理逻辑

### 现有支持的文件类型
- ✅ 图片 (jpeg, png, gif, webp, bmp)
- ✅ PDF
- ✅ Word (doc, docx)
- ✅ Excel (xls, xlsx)
- ✅ 视频 (mp4, mov, avi, mkv, webm) - 已支持但需要确认完整实现
- ❓ 音频 - **待确认是否已支持**

### 现有材料提取策略
| 策略 | 类型 | 说明 |
|------|------|------|
| `invoice` | 发票类 | 8步流程 |
| `structured_doc` | 结构化文档 | OCR + Schema提取 |
| `general_doc` | 通用文档 | OCR + AI分析 |
| `image_only` | 纯图片 | 仅OCR |

### 现有Schema/配置
- `ClaimsMaterial` 类型定义了材料的 JSON Schema 和提取配置
- 支持 `extractionConfig` 包含 `jsonSchema` 和 `aiAuditPrompt`

## 已识别的问题

### 1. OSS URL 时效性问题
**现状**: 
- `ossService.ts` 已有 `getSignedUrl()` 函数
- 但在实际使用中，URL可能在任务处理时过期
- 需要检查当前是否所有地方都使用签名URL而非永久URL

### 2. 异步处理架构
**现状**:
- 已有任务队列系统 (`taskQueue/`)
- 但前端是直接上传base64进行同步分类，再创建异步任务
- 可能导致大量数据在前端传输

### 3. 文件类型支持
**待确认**:
- 音频文件的处理策略
- 视频文件的完整处理流程
- 各种办公文件的解析器

### 4. 材料识别
**现状**:
- 使用 `/api/materials/classify` 进行分类
- 需要确认如何根据材料类型选择对应的 schema

### 5. 结构化提取
**现状**:
- 策略模式已实现
- 需要确认是否所有材料类型都有对应的提取策略

## 关键决策点

### 问题1: 音频文件支持
- **需要确认**: 目前是否有音频文件的处理能力？
- **建议方案**: 添加语音转文字策略

### 问题2: 批量上传流程优化
- **现状**: 前端先同步分类，再异步导入
- **建议方案**: 
  - 选项A: 前端直接批量上传到OSS，后端异步分类+提取
  - 选项B: 保持现状，但优化为批量分类API
  - 选项C: 完全前端处理（不推荐，受网络限制）

### 问题3: OSS URL 处理策略
- **选项A**: 存储OSS Key，每次需要时生成新的签名URL（推荐）
- **选项B**: 存储URL+过期时间，过期时自动刷新

### 问题4: 视频/音频的特殊处理
- 视频: 提取关键帧 + 语音转写
- 音频: 语音转文字
- 如何处理？是否需要新的策略类型？

### 问题5: 错误处理和重试机制
- 如何标记识别失败的文件？
- 是否支持重新识别？
- 失败文件的存储策略？
