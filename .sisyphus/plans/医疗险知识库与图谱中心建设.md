# 医疗险计算方案 - 知识库与图谱中心建设

## 项目概述

**目标**: 根据设计图方案，搭建完整的医疗险知识库与图谱中心，支撑理赔审核的智能化决策

**范围**: 
- 主数据中心建设（药品/诊疗项目/疾病/医院/保单规则/别名映射）
- 规则与知识层建设
- 关系图谱层建设
- Evidence Graph 证据层集成

**技术栈**: Node.js + Express + JSON文件存储（当前）/ PostgreSQL（扩展）

---

## 设计图与现有架构对比分析

### 已有能力 ✅

| 模块 | 现有实现 | 状态 |
|------|----------|------|
| 理赔计算引擎 | `server/claims/settlement/calculator.js` | ✅ 已完整支持医疗险计算 |
| 规则引擎 | `server/rules/engine.js` | ✅ 已支持责任判断和金额计算 |
| AI审核 | `server/ai/` LangGraph | ✅ 已集成计算工具 |
| 产品配置 | `types.ts` 中的 CoveragePlanHealth | ✅ 已定义保额/免赔额/赔付比例 |
| 医疗目录 | `MedicalCatalogManagementPage.tsx` | ⚠️ 基础管理，需扩展 |

### 待建设能力 📋

| 模块 | 设计图要求 | 现有状态 | 优先级 |
|------|------------|----------|--------|
| **主数据中心** | 6张主表（药品/项目/疾病/医院/规则/别名） | ❌ 无 | P0 |
| **规则知识层** | 5类规则（诊断-药品/诊断-项目/剂量/频次/住院必要性） | ⚠️ 部分在理赔引擎中 | P0 |
| **关系图谱层** | 5张关系表 | ❌ 无 | P1 |
| **Evidence Graph** | 案件证据图 | ⚠️ 有auditTrail，需扩展为图结构 | P1 |

### 核心差距

1. **知识库主数据缺失**: 现有系统依赖产品配置，缺少标准化的医疗知识主数据
2. **规则硬编码**: 理赔规则分散在代码中，缺少可配置的规则知识库
3. **关系图谱缺失**: 缺少疾病-药品-项目的关联关系数据
4. **证据链路不完整**: 缺少结构化的审核证据图

## 建设计划分阶段

### 第一阶段：主数据中心（P0 - 核心底座）

#### 1.1 创建数据模型定义文件

**文件**: `server/knowledge/models.js`

**任务**: 定义6张主表的TypeScript/JavaScript类型和验证Schema

**包含内容**:
- `drug_master` 药品主表模型
- `service_item_master` 诊疗项目主表模型
- `disease_master` 疾病主表模型
- `hospital_master` 医院主表模型
- `policy_rule_master` 保单规则主表模型
- `alias_mapping` 别名映射表模型

**参考字段**（来自设计图）:
```javascript
// drug_master 字段
drug_id, generic_name, brand_name, aliases, dosage_form, spec, package, 
manufacturer, nhsa_code, nmpa_approval_no, reimbursement_flag, 
reimbursement_restriction, indications, dose_min, dose_max, 
course_min, course_max, route, source, version, valid_from, valid_to, status
```

#### 1.2 创建数据访问层 (DAO)

**文件**: `server/knowledge/dao/`

**任务**: 
- `drugDao.js` - 药品数据访问
- `serviceItemDao.js` - 诊疗项目数据访问
- `diseaseDao.js` - 疾病数据访问
- `hospitalDao.js` - 医院数据访问
- `policyRuleDao.js` - 保单规则数据访问
- `aliasDao.js` - 别名映射数据访问

**实现要求**:
- 基于现有 `jsonlist/` 目录的JSON文件存储
- 提供CRUD操作
- 支持版本管理和有效期查询

#### 1.3 创建数据导入/ETL模块

**文件**: `server/knowledge/etl/`

**任务**:
- `fileDownloader.js` - 官方文件下载（医保局/药监局）
- `pdfExtractor.js` - PDF表格抽取
- `excelParser.js` - Excel文件解析
- `dataTransformer.js` - 数据清洗和转换
- `versionManager.js` - 版本比对和管理

**目录结构**:
```
/raw
  /nhsa/2024, /nhsa/2025
  /nmpa/2024, /nmpa/2025
  /provincial_mi/shanghai, /beijing
  /hospital
  /internal
```

#### 1.4 创建别名标准化服务

**文件**: `server/knowledge/services/aliasService.js`

**功能**:
- 别名搜索和匹配
- 相似度计算
- 置信度评估
- 标准化映射

**接口**:
```javascript
// 将理赔材料中的原始名称映射到标准实体
normalizeName(rawName, entityType) -> { entityId, entityType, confidence }

// 批量标准化
normalizeBatch(items) -> [{ original, normalized, confidence }]
```

### 第二阶段：规则与知识层（P0 - 审核逻辑）

#### 2.1 创建规则模型定义

**文件**: `server/knowledge/models/rules.js`

**包含规则类型**:
1. **诊断-药品关系规则** (`disease_drug_rule`)
2. **诊断-项目关系规则** (`disease_service_rule`)
3. **剂量/疗程规则** (`dosage_rule`)
4. **项目频次规则** (`frequency_rule`)
5. **住院必要性规则** (`hospitalization_rule`)

**规则表字段**:
```javascript
rule_id, subject_type, subject_id, object_type, object_id, 
rule_type, condition_expr, threshold, action, reason_code, 
priority, source, version, status
```

#### 2.2 创建规则引擎集成层

**文件**: `server/knowledge/ruleEngine.js`

**功能**:
- 加载知识库规则
- 与现有 `server/rules/engine.js` 集成
- 提供规则查询接口

**接口**:
```javascript
// 查询诊断相关的合理性规则
queryReasonabilityRules(diseaseId, itemType) -> [rules]

// 评估医疗合理性
assessMedicalReasonability(diagnosis, treatments) -> assessmentResult
```

#### 2.3 创建医学合理性评估服务

**文件**: `server/knowledge/services/medicalAssessmentService.js`

**功能**:
- 诊断与药品匹配检查
- 诊断与检查项目匹配检查
- 剂量合理性检查
- 频次合理性检查
- 住院必要性评估

**集成点**: 在 `server/rules/engine.js` 的 `executeFullReview` 中调用

---

### 第三阶段：关系图谱层（P1 - 关联分析）

#### 3.1 创建关系模型定义

**文件**: `server/knowledge/models/relationships.js`

**关系表**:
1. `disease_drug_rel` - 疾病-药品关系
2. `disease_service_rel` - 疾病-诊疗项目关系
3. `disease_test_rel` - 疾病-检查关系（可选）
4. `surgery_combo_rel` - 手术组合关系
5. `policy_coverage_rel` - 产品责任覆盖关系

**关系字段**:
```javascript
rel_id, subject_id, object_id, rel_type, evidence_level, 
source, valid_from, valid_to
```

**关系类型枚举**:
- `recommended` - 推荐
- `optional` - 可选
- `not_recommended` - 不推荐
- `contraindicated` - 禁忌
- `required` - 必需（手术组合）

#### 3.2 创建关系查询服务

**文件**: `server/knowledge/services/relationshipService.js`

**功能**:
- 查询疾病相关的推荐药品
- 查询疾病相关的推荐检查
- 查询手术的组合项目
- 查询产品责任覆盖的费用类别

**接口**:
```javascript
getRecommendedDrugs(diseaseId) -> [{ drugId, evidenceLevel }]
getRecommendedServices(diseaseId) -> [{ serviceId, evidenceLevel }]
getSurgeryCombo(surgeryId) -> [{ itemId, required, confidence }]
```

#### 3.3 创建图谱可视化数据接口

**文件**: `server/knowledge/api/graphApi.js`

**功能**: 为前端提供图谱数据查询接口

### 第四阶段：Evidence Graph 证据层（P1 - 审核解释）

#### 4.1 扩展现有审核证据追踪

**文件**: `server/knowledge/evidenceGraph.js`

**节点类型**:
- `CONTRACT` - 合同节点
- `DIAGNOSIS` - 诊断节点
- `DRUG` - 药品节点
- `SERVICE` - 项目节点
- `AMOUNT` - 金额节点
- `RULE` - 规则节点
- `MODEL` - 模型节点
- `CONCLUSION` - 结论节点

**关系类型**:
- `supports` - 支持
- `contradicts` - 矛盾
- `triggers` - 触发
- `derived_from` - 派生
- `exceeds_threshold` - 超出阈值
- `matches_policy` - 符合保单

#### 4.2 集成到理赔审核流程

**修改文件**: `server/rules/engine.js`

**任务**: 在 `executeFullReview` 中构建Evidence Graph

**输出示例**:
```json
{
  "caseId": "CASE001",
  "nodes": [...],
  "edges": [...],
  "conclusion": {
    "decision": "MANUAL_REVIEW",
    "reason": "轻症高阶检查存疑",
    "riskScore": 0.82
  }
}
```

#### 4.3 创建证据图可视化服务

**文件**: `server/knowledge/services/evidenceVisualizationService.js`

**功能**: 将证据图转换为前端可展示的格式

## 技术风险与缓解措施

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 官方数据格式变化 | 高 | 保留原始文件，版本化管理 |
| 别名映射覆盖不全 | 中 | 建立人工反馈机制，持续优化 |
| 规则配置复杂 | 中 | 提供可视化规则编辑器 |
| 性能瓶颈（大量关系查询） | 中 | 考虑引入缓存和索引 |
| 数据一致性 | 高 | 建立数据校验和审计机制 |
