# AI 智能理赔系统功能点整理

## 功能点清单

| 功能大类 | 功能点 | 功能描述 | 实现状态备注 |
|----------|--------|----------|--------------|
| 智能申请 | 资料智能分类与齐全性提示 | 基于客户录入的出险信息，系统自动匹配需上传的资料类别。客户完成拍摄或上传后，资料实时自动分类并按细类标记展示，同步提示理赔资料齐全性，提升客户一次性提交通过率。涵盖资料类别包括：身份证明类（身份证、户口簿等）、事故证明类（病历、意外事故证明等）、医疗票据类（发票、费用明细清单等）、保司文件类（保险合同、索赔申请书等）、银行账户类（银行卡、存折等） | **已实现** - `services/material/materialClassifier.ts` 实现了AI材料分类；`server/claims/materials/completeness.js` 实现了材料齐全性检查；支持文档上传后的自动分类和提示。 |
| 智能申请 | 影像质量管控 | 支持识别影像复印件/翻拍件，具备影像纠正/纠偏、模糊识别等功能，模糊影像实时发出提示，确保影像资料符合合规要求，为后续智能采集环节提供高质量影像支撑 | **部分实现** - `materialClassifier.ts` 有模糊度检测（confidence < 0.5视为模糊）；复印件/翻拍件识别、影像纠偏功能尚未明确实现。 |
| 智能申请 | AI 初审功能构建 | 依据初审签收规范，开发 AI 初审功能，对保单有效性、出险日期在保障期间、单证齐全性、基本信息提供完整性等初审维度进行自动化校验 | **已实现** - `server/ai/agent.js` 和 `server/rules/engine.js` 实现了责任判断；`evaluateMaterialCompleteness` 实现材料齐全性校验；支持保单有效性、保障期间等初审。 |
| 智能申请 | 初审结论与报告生成 | AI 基于采集后的案件信息，自动生成初审结论与专项报告，为后续处理环节提供清晰依据 | **已实现** - `server/rules/engine.js` 的 `executeFullReview` 生成完整审核报告；包含责任判断、金额计算、缺失材料等结论；支持 `APPROVE/REJECT/MANUAL_REVIEW` 决策。 |
| 智能采集 | 数据提取 | 根据案件处理信息需求，对相关影像信息进行分类、关联与数据提取，提取后进行标准化处理。提取内容包括：客户基本信息、发票、费用明细清单、结算单、出院小结、病案首页、检查报告等。作为案件决策和相关风险模型的基础数据提取 | **已实现** - `services/invoiceOcrService.ts` 实现医疗发票OCR识别；`services/material/` 下策略模式实现不同材料类型的数据提取；支持发票、病历、诊断证明等多种材料。 |
| 智能采集 | 数据标准化处理 | 依据预设采集规则，实现影像提取信息与非影像信息的各类字段值的规范化校验、对码等，确保后续数据使用的规范化和有效性。采集规则包括公司内部采集规则（如医保扣费规则、第三方给付采集规则等），以及各类数据对码与标准化治理（如司内数据对码、医保三目录、ICD 编码、ICD-O-3 编码、手术操作代码、医院等国家或行业编码的对码及标准化治理） | **部分实现** - `services/catalogMatchService.ts` 实现医保三目录匹配和标准化；`server/services/documentParser.js` 实现文档解析；ICD编码、手术操作代码对码尚未明确实现。 |
| 智能采集 | 事件梳理与影像关联 | 按照既定规则，基于影像梳理生成案件事件与就诊时间线，并从事件、就诊、账单三个维度完成影像关联，关联影像涵盖病历、发票、费用明细清单、结算单等关键材料 | **部分实现** - `utils/claimTimelinePresentation.ts` 实现理赔流程时间线展示；`ClaimCaseDetailPage.tsx` 支持影像文件分类展示；完整的事件-就诊-账单三维关联尚未实现。 |
| 智能采集 | 模型交叉验证 | 为加强数据采集准确性，使用模型交叉验证机制 | **未实现** - 当前OCR识别使用单一模型（Gemini/GLM/PaddleOCR可选），但模型间的交叉验证机制尚未实现。 |
| 智能决策-免责风险识别 | 责免条款解析与识别 | 解析全部医疗产品中的责免条款（包括但不限于既往症、精神和行为障碍、康复治疗、牙齿治疗、分娩等），结合知识图谱与案件材料，识别案件中需拒付的责任范围，标记责免涉及需剔除的治疗费用 | **部分实现** - `server/rules/engine.js` 和 `evaluator.js` 实现责任判断；`checkEligibilityTool.js` 支持除外责任检查（如酒驾、等待期）；知识图谱结合尚未实现；既往症等医疗责免需结合病历分析能力。 |
| 智能决策-免责风险识别 | 特约疾病识别 | 利用新技术，对于匹配理算对应险种中的特别约定文字内容，识别特约疾病，根据费用明细清单剔除特约疾病涉及的治疗费用或提示拒付 | **未实现** - 当前系统主要基于通用责任规则判断，针对特定疾病的特约条款识别和费用剔除逻辑尚未专门实现。 |
| 智能决策-免责风险识别 | 责免判断依据 | 上述所有判断均基于当前赔案匹配理算所涉及的保单险种对应的责免条款与特别约定 | **部分实现** - `server/rules/` 目录下规则引擎支持基于产品条款的责任判断；`jsonlist/rulesets.json` 存储规则集；但责免条款的精细化解析和专用判断逻辑需要增强。 |
| 智能决策-健康险精准提调模型 | 风险案件学习 | 通过学习历史风险案件数据（含保单信息、索赔材料影像、历史赔案全部数据），结合公司理赔实务规则规范，总结提炼调查提调的类型与核心原因 | **未实现** - 无历史案件学习机制；无风险案件数据训练模型；调查提调类型和原因的AI学习功能尚未实现。 |
| 智能决策-健康险精准提调模型 | 调查发起判断 | AI 依据学习形成的提调方法，判断当前案件是否需要发起调查，并输出发起调查原因和建议调查方向 | **部分实现** - `server/rules/evaluator.js` 有 `fraudFlagged` 欺诈风险标记；`fraudRiskScore` 风险评分；但基于历史学习的精准提调判断尚未实现。 |
| 智能决策-医疗案件AI预审 | AI 预审智能体构建 | 整合智能申请中的初审、免责风险识别、健康险精准提调模型，结合公司风控规则/模型/数据以及审核要求，构建 AI 预审智能体，输出赔案预审结论与审核思维过程 | **已实现** - `server/ai/agent.js` + `server/ai/graph.js` LangGraph实现；`server/rules/engine.js` 规则引擎；输出 `decision/reasoning/eligibility/calculation` 完整审核结论。 |
| 智能决策-医疗案件AI预审 | 模型自学习 | 支持基于新的案件数据或新增理赔相关文件进行自学习，通过业务人员提供错误分析或案件审核意见进行模型微调，持续提升 AI 预审结论的准确性 | **未实现** - 当前使用预定义Prompt和规则引擎，无模型自学习和微调机制；无基于反馈的错误修正学习。 |
| 智能决策-医疗案件AI预审 | 病历材料读取与分析 | AI 自动读取入/出院记录、门诊病历等材料，根据病历中载明的疾病情况，按时间轨迹列示关键健康信息（包括但不限于既往史、现病史、遗传史、重点指标异常指征、治疗经过） | **部分实现** - `services/invoiceOcrService.ts` 可识别诊断、出院小结等；`services/material/` 策略模式支持病历材料解析；但完整的病历时间轨迹分析和健康信息提取尚未实现。 |
| 智能决策-医疗案件AI预审 | 病历标准化提炼 | 对于病历中的长段文字由 AI 进行标准化提炼，以符合展示需求和后续数据使用 | **未实现** - 当前OCR提取后直接展示原始文本，无专门的长段文字标准化提炼和结构化处理功能。 |
| 智能决策-医疗案件AI预审 | 客户健康画像构建 | 结合历史赔案记录，构建客户健康画像 | **未实现** - 无客户健康画像模块；无历史赔案数据关联分析；无客户健康信息聚合展示功能。 |
| 智能决策-重疾决策智能体 | 重疾知识图谱构建 | 依据公司重疾相关条款标准定义，分析不同险种、不同病种的理赔适用范围、判断原则、判断方法与标准，结合理赔重疾病种核定的实际业务需求，搭建医疗知识图谱（例如内置"糖尿病→糖尿病肾病→终末期肾病"等疾病关联网络，自动验证并发症逻辑链；关联医疗权威文献） | **未实现** - 有 `schemas/critical_illness_product_schema.json` 重疾产品配置Schema；但无知识图谱实现；无疾病关联网络和并发症逻辑链验证。 |
| 智能决策-重疾决策智能体 | AI 推理病种核定 | 基于当前赔案相关数据（包括但不限于赔案相关影像材料及结构化数据、保单信息、险种信息、赔案信息等）与重疾知识图谱，依据重疾预审算法，实现重疾病种责任核定的自动化、标准化输出 | **未实现** - 当前理赔审核主要针对医疗险；无重疾专用的病种核定算法；无结合知识图谱的重疾推理能力。 |
| 智能决策-重疾决策智能体 | 核定报告生成 | 系统自动生成详细的重疾病病种核定报告，支持根据需求定制输出格式 | **未实现** - 无重疾专用的核定报告生成功能；当前 `executeFullReview` 生成通用审核报告，非针对重疾的详细核定报告。 |
| 理赔全流程监控平台 | 指标统计 | 覆盖理赔全流程（报案、签收、采集/立案、核批、结案支付），统计整体及各环节的自动通过率、处理时效，以及各类模型的应用效能（如正确率），为识别流程异常、优化 AI 效能提供数据支撑，最终实现自动通过率与 AI 处理正确率的提升 | **部分实现** - `ClaimCaseListPage.tsx` 有基础统计（全部/待处理/处理中/累计索赔额）；`SystemLogsPage.tsx` 有AI审核日志记录；但**自动通过率、处理时效、模型效能**等深度统计指标未实现。 |
| 理赔全流程监控平台 | 指标展示 | 搭建独立的自动化效能看板，支持实时/日/周/月/年数据展示，看板内容包括但不限于实施效果指标。准确率指标需根据实际作业情况与作业监督需求进行分类细化，支持依据作业系统反馈的提取错误，及时开展模型迭代与调优 | **部分实现** - `DataDashboardPage.tsx` 有通用数据看板；`adjuster-workbench/src/pages/DataStats.tsx` 有AI助手统计；但**理赔专用的自动化效能看板**（含自动通过率、准确率等）未实现。 |
| 理赔全流程监控平台 | 异常监控 | 对达到指定阈值的异常情况（如某环节通过率过低、识别错误率过高、影像识别环节排队拥堵等）进行实时提示预警，并支持手动调整任务队列，确保流程顺畅运行 | **未实现** - 无异常监控预警机制；无阈值设定和告警功能；无任务队列管理功能。 |
| 模型自学习与动态调优 | 模型自学习 | 支持模型基于新的案件数据或新增理赔相关文件进行自学习 | **未实现** - 当前AI模型使用固定Prompt和规则；无增量学习机制；无从新案件数据中提取知识并更新模型的能力。 |
| 模型自学习与动态调优 | 模型动态调优 | 通过业务人员提供错误分析或案件审核意见进行模型微调，持续提升 AI 预审结论的准确性 | **未实现** - 无反馈闭环机制；无基于人工审核意见的模型微调能力；无持续学习优化系统。 |
| 项目上线与全链路保障 | 项目上线方案 | 提供从机构试点到全国推广的完整项目上线方案 | **未实现** - 当前为开发/测试系统；无完整的机构试点到全国推广的上线方案和流程设计。 |
| 项目上线与全链路保障 | 全链路性能监控 | 部署实时监控系统，对核心指标进行持续追踪，关键监控指标包括：模型推理延迟 P99（99% 请求的响应时间）≤500ms；异常样本识别率等 | **未实现** - 有基础的AI审核耗时记录；但无全链路性能监控系统；无P99延迟统计、异常样本识别率等专业监控指标。 |
| 项目上线与全链路保障 | 生产环境故障应急响应 | 制定完善的生产环境故障应急响应机制，明确故障识别、上报、处置流程，确保突发问题快速响应、及时解决 | **未实现** - 当前为开发环境；无生产环境故障应急响应机制；有 `deploy.sh` 部署脚本和 `ecosystem.config.cjs` PM2配置，但无故障分级处理和应急预案。 |
| 项目上线与全链路保障 | 故障分级处理 | 一级故障（重大故障）≤15分钟响应，切换备用系统，7×24小时人工介入；二级故障（功能异常）≤1小时响应，限流降级，4小时内修复；三级故障（性能下降）≤24小时响应，优化资源分配，7日内恢复 | **未实现** - 无故障分级体系；无备用系统切换机制；无7×24小时运维支持；无明确的故障响应时效承诺。 |
| 知识体系动态更新 | 知识体系本地化部署 | 案件全流程处理所依赖的知识体系需实现本地化部署，确保数据安全、合规与业务灵活性 | **部分实现** - 系统支持本地化部署（Docker + PM2）；医保三目录、产品条款等数据存储于本地JSON；但知识更新机制需人工维护，非自动化更新。 |
| 知识体系动态更新 | 知识覆盖全面性 | 通过动态更新机制确保知识体系覆盖完整，无关键领域遗漏 | **部分实现** - 有产品条款、医保三目录等基础数据；但知识更新依赖人工导入；无自动化的知识覆盖检查和补全机制。 |
| 知识体系动态更新 | 逻辑一致性保障 | 借助数据治理与验证流程，确保知识体系内部逻辑统一、无矛盾 | **未实现** - 无自动化的知识一致性校验；无数据治理流程；无知识冲突检测机制。 |
| 知识体系动态更新 | 数据分级动态管理 | 对知识数据实施分级动态管理 | **未实现** - 无知识数据分级体系；无动态管理策略；所有知识数据同等对待。 |
| 知识体系动态更新 | 知识体系范围 | 包括但不限于：医保三目录、产品条款、初审标准、数据采集标准、医疗决策审核标准、重疾病种判定标准、医疗知识 | **部分实现** - 已实现：医保三目录（`catalogMatchService.ts`）、产品条款（`jsonlist/` 产品数据）、初审标准/数据采集标准（`services/material/`）；**未实现**：医疗决策审核标准、重疾病种判定标准、完整医疗知识体系。 |
| 项目全周期优化支持 | 项目实施期支持 | 在项目实施过程中，需针对新产品上线、新业务场景拓展提供及时的功能优化支持，确保系统适配业务发展需求 | **不适用** - 此为项目管理/商务要求，非技术功能点。系统已支持产品配置化扩展（`ProductConfigPage.tsx`）。 |
| 项目全周期优化支持 | 项目运维期支持 | 在项目运维阶段，需支持项目范围内的功能改造、核心指标提升等优化需求，同时可按项目范围内运维承接项目外简单功能改造需求 | **不适用** - 此为项目管理/商务要求，非技术功能点。系统基于模块化架构，支持功能扩展和指标优化。 |
| 非功能需求 | 接口性能要求 | 平均响应时间 ≤500 毫秒，最大响应时间 ＜2 秒 | **待验证** - 系统使用AI模型(Gemini)推理，实际响应时间受模型影响；需进行性能测试验证是否满足要求。 |
| 非功能需求 | 影像分类要求 | 单案件（影像量 20-30 张）响应时间 ≤3 秒 | **待验证** - `materialClassifier.ts` 使用AI分类，单张影像需调用LLM；并发处理下需验证是否能满足3秒要求。 |
| 非功能需求 | 基础软件要求 | 操作系统、中间件及数据库需满足全栈信创标准 | **待验证** - 当前使用Node.js + Express + React技术栈；需确认是否满足信创要求（国产操作系统、数据库等）。 |

| 非功能需求 | 架构特性要求 | 技术架构需具备高可用、易扩展能力，并需提供完整的运维保障功能 | **部分实现** - 使用模块化架构（components/services/server分层）；支持Docker部署；PM2进程管理；但高可用集群、负载均衡需额外配置。 |
